Anand Rathi × FieldProxy — CRM Integration Guide

One place for the database schema, REST endpoints and ready-to-paste sample requests for the ARSSBL ↔ FieldProxy CRM integration: the customer (lead) master and the visit master captured by field reps.

v1.0 · 2026 Tenant · arssbl Base URL · https://api-india-1.fieldproxy.ai Auth · x-api-key (JWT)

Overview

FieldProxy is the operational database for ARSSBL's field CRM. Two tables are in scope:

  • customer_master_sheet — the customer / lead master: one row per customer with type, stage, contact, owner rep and location.
  • visit_master_sheet — one row per field visit (check-in / check-out with GPS, purpose, remarks and duration). Each visit links to customer_master_sheet.customerid.

Every change is a single REST call against FieldProxy's /api/insert, /api/update, /api/read, or /api/delete endpoint with an x-api-key header. Bulk loads use the encrypted /api/import-csv endpoint.

REST endpoints

Bulk-insert one or many rows into a table.

POST/api/read

Select rows by an exact-match condition.

POST/api/update

Update column values on rows matching a condition.

POST/api/delete

Delete rows matching a condition.

Encrypted bulk import AES-256-CBC

For large loads (initial customer / visit migration, month-end batches), data is sent as an AES-256-CBC encrypted CSV via POST /api/import-csv. ARSSBL encrypts the CSV with the shared key + IV and base64-encodes it; FieldProxy decrypts it using the key/IV held in its tenant secrets vault, validates every row against the live table schema, and inserts all rows in one atomic statement.

The key & IV are never sent in the request. The body carries only the names of the vault secrets (keySecret / ivSecret). There is no per-request password — the key/IV are provisioned once during setup.

Encryption recipe

ParameterValue
AlgorithmAES-256-CBC
PaddingPKCS7 (PKCS5 for AES in Java)
Key32 bytes, base64-encoded
IV16 bytes, base64-encoded
Ciphertext (file)base64-encoded
CSVUTF-8 · row 1 = column names, matching the target table exactly

Request

curl --request POST 'https://api-india-1.fieldproxy.ai/api/import-csv' \
  --header 'x-api-key: <YOUR_API_KEY>' \
  --header 'Content-Type: application/json' \
  --data '{
    "tableName": "customer_master_sheet",
    "file": "<base64 of the AES-256-CBC ciphertext>",
    "keySecret": "CRM_IMPORT_KEY",
    "ivSecret": "CRM_IMPORT_IV"
  }'

Success response

{
  "message": "Successfully imported 250 rows into customer_master_sheet",
  "rowCount": 250,
  "table": "customer_master_sheet",
  "tenant": "arssbl",
  "columns": ["customerid", "customer_name", "customer_stage", "..."]
}

On validation failure the endpoint returns 400 with a stage (secrets, decrypt, csv, columns, rows, insert) so you know exactly where it failed. The full encryption recipe (Node / Python / Java) is on the HR guide — the encryption is identical.
