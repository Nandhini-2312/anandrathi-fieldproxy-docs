1. Overview & Conventions

This document specifies the APIs Field Proxy exposes so the Anand Rathi CRM system can exchange customer (lead) and visit data with Field Proxy. It mirrors the E-HR Integration API Document — the same authentication and AES-256 conventions apply. Customer and visit records flow in via the endpoints below; the same records are available to be read back for two-way sync.

Global conventions apply to every endpoint:

FieldValue
Base URLhttps://api-anandrathi.fieldproxy.ai  (per env: api-anandrathi.sit / .uat / .fieldproxy.ai)
TransportHTTPS only (TLS 1.2+). Endpoints are served ONLY on the host above.
EncryptionAES-256-CBC, PKCS7 padding, Base64. Key = 32 bytes, IV = 16 bytes; issued by Field Proxy per environment.
Request bodyapplication/json  →  { "data": "<Base64 ciphertext>" }
AuthorizationAuthorization: Bearer <JWT>  (from /crm/auth). Token is valid 1 minute.
Response envelope{ result, resultMessage, resultData }
Result enum0 = Error, 1 = Success, 2 = Info, 3 = Warning, 4 = Attention
Batch + partial successInsert/Update/Upsert accept a JSON array; each object is validated independently; valid rows apply, invalid rows are reported (never a whole-batch rollback).

3. Authentication API

FieldValue
TitleField Proxy CRM Authentication API
Request URLPOST  https://api-anandrathi.fieldproxy.ai/crm/auth
MethodPOST
HeadersContent-Type: application/json
BodyAES-256 encrypted + Base64 of the request model, wrapped as { "data": "…" }

Request model (decrypted)

{
  "Username": "string",
  "Password": "string",
  "AppCode":  "string"
}

Sample (decrypted, then wire body)

{ "Username": "AnandRathi_CRM", "Password": "********", "AppCode": "FPAPP1001" }

// on the wire:
{ "data": "<AES-256-CBC/PKCS7/Base64 of the JSON above>" }

Responses

// success
{ "result": 1, "resultMessage": "Token generated successfully", "resultData": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." }

// invalid credentials
{ "result": 3, "resultMessage": "Username or password is invalid", "resultData": null }

// error
{ "result": 0, "resultMessage": "Exception message", "resultData": null }
  • resultData is the JWT Bearer token; valid 1 minute. Re-authenticate immediately before each Customer/Visit call.
  • The same auth mechanism as the E-HR API; a single AppCode may be issued for both CRM and E-HR, or scoped separately.

13. Security

  • HTTPS only; the partner endpoints are served exclusively on api-anandrathi.fieldproxy.ai and are IP-allowlisted to Anand Rathi CRM source ranges.
  • AES-256 key/IV and partner credentials are held in Field Proxy's encrypted secret store, referenced by name; rotated per environment.
  • A VAPT (security pen-test) will be completed before production, targeting these inbound endpoints.

Recommended crypto hardening (for both directions):

  • Use a per-message random IV prepended to the ciphertext instead of a fixed shared IV — a static IV in CBC leaks equality of identical plaintexts.
  • Add a MAC (or move to AES-256-GCM) so tampering is detected; CBC alone is unauthenticated.

14. Notes & Open Items

  • Field names above are the proposed API contract; the final customer/visit field list is to be confirmed with Anand Rathi CRM (additional columns can be added).
  • Keys, credentials and real base URLs are exchanged per environment (SIT / UAT / PROD) during onboarding.
  • Two-way sync: the same customer/visit records are available to be read back (rep edits pushed from Field Proxy). A read/pull endpoint can be added on the same conventions if required.
  • resultData carries structured counts + per-row errors so the CRM can reconcile exactly which rows applied; it can be reduced to the minimal shape on request.
