Anand Rathi × FieldProxy — Integration Guide

One place for the database schema, REST endpoints and ready-to-paste sample requests for the ARSSBL ↔ FieldProxy integration: daily employee attendance (punch-in / punch-out with selfie + GPS), the user master synced from the E-HR system, and the team master.

v1.0 · 2026 Tenant · arssbl Base URL · https://api-india-1.fieldproxy.ai Auth · x-api-key (JWT)

Overview

FieldProxy is the operational database for ARSSBL's field workforce. Three tables are in scope:

  • attendance_ukrwi — one row per employee per working day. Records punch-in / punch-out selfie, GPS coordinates, time, address, and computed working hours.
  • fp_users — master list of users who can log in to the FieldProxy mobile / web app. ARSSBL pushes this whenever a user is created, updated, or deactivated in the E-HR system.
  • fp_teams — master list of teams / groups that users are assigned to. Each team links to fp_users.teamid.

Every change is a single REST call against FieldProxy's /api/insert, /api/update, /api/read, or /api/delete endpoint with an x-api-key header.

Encrypted bulk import AES-256-CBC

For large loads (e.g. month-end attendance), data is sent to FieldProxy as an AES-256-CBC encrypted CSV via POST /api/import-csv. ARSSBL encrypts the CSV with the shared key + IV and base64-encodes it; FieldProxy decrypts it automatically using the key/IV held in its tenant secrets vault, validates every row against the live table schema, and inserts all rows in one atomic statement — if any row is invalid, nothing is inserted.

The key & IV are never sent in the request. The body carries only the names of the secrets (keySecret / ivSecret) that FieldProxy holds in its vault. There is no per-request password — the key/IV are provisioned once during setup.

Encryption recipe

ParameterValue
AlgorithmAES-256-CBC
PaddingPKCS7 (PKCS5 for AES in Java)
Key32 bytes, base64-encoded
IV16 bytes, base64-encoded
Ciphertext (file)base64-encoded
CSVUTF-8 · row 1 = column names, matching the target table exactly

curl --request POST 'https://api-india-1.fieldproxy.ai/api/import-csv' \
  --header 'x-api-key: <YOUR_API_KEY>' \
  --header 'Content-Type: application/json' \
  --data '{
    "tableName": "attendance_ukrwi",
    "file": "<base64 of the AES-256-CBC ciphertext>",
    "keySecret": "EHR_IMPORT_KEY",

Validation failures (HTTP 400, all-or-nothing)

The body always includes a stage telling you where it failed:

stageMeaning
secretsUnknown secret name, or wrong key/IV length.
decryptWrong key/IV or corrupted ciphertext.
csvEmpty, header-only, or too many rows.
columnsUnknown / duplicate / empty header columns (response lists the valid columns).
rowsPer-row errors {row, column, value, message} (capped at 100).
insertMapped Postgres constraint error (e.g. unique violation).

Other codes: 401 (API key required), 403 (key not authorized for insert / this table), 429 (write rate limit exceeded), 500.

Building the encrypted file

Steps: (1) build the CSV with row 1 = exact column names; (2) AES-256-CBC encrypt the bytes with the shared 32-byte key and 16-byte IV (PKCS7); (3) base64-encode the ciphertext → that string is the file field.

Node.js 
const crypto = require('crypto');

// the agreed values (base64), the SAME ones vaulted in FieldProxy
const key = Buffer.from(process.env.EHR_IMPORT_KEY_B64, 'base64'); // 32 bytes
const iv  = Buffer.from(process.env.EHR_IMPORT_IV_B64,  'base64'); // 16 bytes

function encryptCsv(csvString) {
  const cipher = crypto.createCipheriv('aes-256-cbc', key, iv); // PKCS7 by default
  const enc = Buffer.concat([cipher.update(csvString, 'utf8'), cipher.final()]);
  return enc.toString('base64'); // -> goes in "file"
}
Python 
import base64
from Crypto.Cipher import AES
from Crypto.Util.Padding import pad

key = base64.b64decode(EHR_IMPORT_KEY_B64)  # 32 bytes
iv  = base64.b64decode(EHR_IMPORT_IV_B64)   # 16 bytes

def encrypt_csv(csv_string: str) -> str:
    cipher = AES.new(key, AES.MODE_CBC, iv)
    ct = cipher.encrypt(pad(csv_string.encode('utf-8'), AES.block_size))  # PKCS7
    return base64.b64encode(ct).decode()    # -> goes in "file"
Java 
SecretKeySpec k = new SecretKeySpec(Base64.getDecoder().decode(KEY_B64), "AES"); // 32 bytes
IvParameterSpec ivp = new IvParameterSpec(Base64.getDecoder().decode(IV_B64));   // 16 bytes
Cipher c = Cipher.getInstance("AES/CBC/PKCS5Padding"); // PKCS5 == PKCS7 for AES
c.init(Cipher.ENCRYPT_MODE, k, ivp);
String file = Base64.getEncoder()
    .encodeToString(c.doFinal(csv.getBytes(StandardCharsets.UTF_8)));
Decryption is automatic on FieldProxy's side. ARSSBL never calls a decrypt step — FieldProxy resolves the vaulted key/IV, decrypts, validates and inserts.

One-time key & IV setup

  1. Generate a 32-byte key and 16-byte IV (base64): 
    openssl rand -base64 32   # KEY (32 bytes)
openssl rand -base64 16   # IV  (16 bytes)
  2. FieldProxy stores them in the tenant vault as EHR_IMPORT_KEY / EHR_IMPORT_IV.
  3. The same key/IV are shared with ARSSBL out-of-band for the encrypt side.
Share the key/IV out-of-band (not over email). To rotate, update both the vault secret and the ARSSBL encrypt side together.
